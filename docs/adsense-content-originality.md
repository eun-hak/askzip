# 애드센스 콘텐츠 독창성 점검 가이드

애스크집(askzip.kr) 안내글의 Google AdSense 심사 관점 **「가치가 낮은 콘텐츠 / 유사문서」** 리스크를 정리한 문서입니다.

**점검 기준일:** 2026년 6월 16일  
**대상:** `src/data/articles/*.ts` (약 40개 안내글)

---

## 1. Google이 보는 핵심 원칙

AdSense·게시자 정책에서 특히 강조되는 항목은 다음과 같습니다.

| 항목 | 요구 |
|------|------|
| **독창성** | 정부·공공기관 안내를 그대로 복사한 페이지가 아닌, **본인만의 요약·정리·실용 정보** |
| **가치** | 사용자가 실제 절차를 진행할 때 도움이 되는 **단계별 안내, 표, FAQ, 주의사항** |
| **신뢰** | 공식 출처 링크, 면책·운영자 정보, 개인정보처리방침(광고 쿠키 고지) |
| **중복** | 여러 URL에 **동일 문단·동일 면책 문구**가 반복되면 유사/저가치 페이지로 분류될 수 있음 |

> **가장 중요:** 공공기관 홈페이지 문구를 붙여넣기만 한 형태는 **「유사문서」**로 거절될 수 있습니다.  
> 깔끔한 요약, 실제 진행 시 주의사항, 자주 묻는 질문(FAQ) 등으로 **독창성을 확보**해야 합니다.

---

## 2. 현재 사이트 강점 (유지할 것)

점검 시 확인된 **잘 된 부분**입니다. 신규 글·수정 시에도 이 패턴을 유지하세요.

### 2.1 구조·형식

- 단계별 가이드(`steps`), 요약 박스(`summary`), 목차(`toc`)
- 표(`table`)로 옵션·요건·기간 정리
- `tip` / `notice` / `warning` 박스로 실용 정보 구분
- FAQ 블록 + JSON-LD `FAQPage` (크롤러·검색 노출)

### 2.2 내용 측면

- 2026년 기준 소득 상향 등 **시점이 있는 정보** 반영 (예: 근로장려금 맞벌이 4,400만원)
- 발급 옵션 선택 기준 (예: 주민등록등본 뒷자리 비표시 권장)
- 관청 FAQ에 없는 **실무형 FAQ** 일부 포함 (예: 연납 후 차량 매도 환급, 부모 재산 합산)

### 2.3 사이트 전반

- 사이트 목적·비연계 고지 (정부·공공기관과 무관한 개인 운영 사이트)
- 공식 출처 링크(`SourcesBox`)
- 개인정보처리방침 내 AdSense·쿠키 고지

## 3. 리스크 항목 (수정 필요)

### 3.1 ★★★ `warning` 박스 문구 전 글 동일 (최우선)

**문제**

거의 모든 안내글 하단에 아래와 **같은 패턴**의 면책 문구가 반복됩니다.

```
이 사이트에서 제공하는 정보는 참고 목적으로 작성되었습니다.
정확한 절차 및 최신 정보는 [기관] 공식 사이트(URL)에서 확인하시기 바랍니다.
```

Google은 **여러 페이지에 동일한 마무리 문단**이 반복되는 것을 중복·저가치 콘텐츠 신호로 볼 수 있습니다.

**조치**

- 글마다 **해당 주제·기관·실수 포인트**에 맞는 warning 1문단으로 교체
- 공식 URL만 바꾼 복붙은 피하고, **그 글에서만 의미 있는 주의**를 넣기

**교체 예시**

| 글 주제 | warning 예시 |
|---------|----------------|
| 근로장려금 | 장려금 지급액과 신청 결과는 국세청 내부 심사 후 결정됩니다. 안내문 수령이 곧 지급 확정이 아닙니다. |
| 자동차세 연납 | 연납 할인율은 매년 지방세 조례에 따라 변동됩니다. 1월 연납 기간 시작 전 위택스 공지를 확인하세요. |
| 주민등록등본 | 제출처가 요구하는 항목(뒷자리 표시, 세대원 전원 등)을 먼저 확인한 뒤 발급 옵션을 선택하세요. |

**작업 위치:** 각 `src/data/articles/*.ts`의 `type: 'warning'` 블록

---

### 3.2 ★★☆ FAQ가 예측 가능한 패턴 반복

**문제**

다수 글에서 FAQ가 다음 패턴으로만 구성된 경우가 많습니다.

- 수수료 있나요?
- 대리 발급 / 대리인 가능?
- 유효기간 있나요?

공공기관 사이트 FAQ와 겹치면 **독창성 점수가 낮아집니다.**

**조치**

- 글당 FAQ **1~2개**를 「실패·막히는 상황」 질문으로 추가
- 관청 홈페이지에 없는 **절차 중 오류·시스템 이슈**를 다루기

**추가 FAQ 예시**

| 글 | 질문 | 답변 요지 |
|----|------|-----------|
| 주민등록등본 | 발급 버튼이 비활성화됩니다 | 점검 시간, 팝업 차단(PDF), 비회원 인증 실패 |
| 건강보험 자격득실 | 퇴사 후 상실이 안 보입니다 | 회사 자격 상실 신고 지연(퇴사 후 2~3주) |
| 홈택스 지급명세서 | 조회 결과가 없습니다 | 회사 제출 전, 귀속 연도·제출 기한(3월 이후) |

**작업 위치:** 각 글 `type: 'faq'` → `items` 배열

---

### 3.3 ★☆☆ 일부 글 본문 두께 부족

**문제**

스텝 4~5개 + FAQ 3개 수준으로 다른 글 대비 얇은 경우가 있습니다.

**우선 보강 후보**

| slug | 제목 | 보강 방향 |
|------|------|-----------|
| `hometax-payment-statement` | 홈택스 지급명세서 조회 | 소득확인증명서와 차이, 조회 안 될 때 체크리스트, 활용 사례(대출·장학금) |
| `telecom-unrefunded-fee` | 통신비 미환급금 | 통신사별 앱 경로, 소멸 시한, 정부24 통합 조회 한 줄 요약 |

**권장 최소 구성 (안내글 1편)**

- 본문 섹션 4개 이상 (`h2`)
- 표 또는 `tip`/`notice` 2개 이상
- FAQ 4개 이상 (실패 케이스 1개 이상 포함)
- 글 전용 `warning` 1개

## 4. 글 작성·수정 체크리스트

새 글 추가 또는 기존 글 수정 시 아래를 확인하세요.

### 독창성

- [ ] 공공기관 문구를 **문장 단위로 복사**하지 않았는가?
- [ ] 「왜 / 언제 / 어떤 옵션을」 선택하는지 **판단 기준**이 있는가?
- [ ] FAQ에 **실제로 막히는 상황** 질문이 1개 이상 있는가?
- [ ] `tip` / `notice`에 **관청 페이지에 없는 실무 팁**이 있는가?

### 중복·품질

- [ ] `warning` 문구가 **다른 글과 동일하지 않은가?**
- [ ] 면책만 있고 본문이 절차 나열만 하지 않는가?
- [ ] 공식 출처 링크·확인일이 있는가? (`SourcesBox` / `articleSources`)

### 기술·SEO (사이트 공통)

- [ ] FAQ 답변이 초기 HTML에 포함되는가? (`<details>` / `FAQBlock`)
- [ ] 표는 `<table>` 구조로 제공되는가? (`ArticleTable`)
- [ ] 관련 글(`relatedSlugs`) 2개 연결

## 6. `_template.ts` 권장 패턴

새 안내글 템플릿에 반영할 블록 타입 예시입니다.

```ts
// 글 주제에 맞는 notice (최신 변경·놓치기 쉬운 점)
{ type: 'notice', text: '…' },

// 실무 팁 (관청 페이지에 없는 선택 기준)
{ type: 'tip', text: '…' },

// 글마다 다른 warning (동일 면책 문구 금지)
{ type: 'warning', text: '…' },

// FAQ: 일반 질문 + 실패/오류 케이스 1개 이상
{
  type: 'faq',
  items: [
    { question: '…', answer: '…' },
    { question: '○○이 안 될 때는?', answer: '…' },
  ],
},
```

**금지 패턴**

```ts
// ❌ 모든 글에 동일
{ type: 'warning', text: '이 사이트에서 제공하는 정보는 참고 목적으로 작성되었습니다. …' }
```

## 7. 애드센스 심사와의 관계 (요약)

| 심사 관점 | 애스크집 현재 상태 | 권장 |
|-----------|-------------------|------|
| 콘텐츠 양·카테고리 | 40개+, 4카테고리 — 양호 | 유지·보강 |
| 사이트 목적·문의·개인정보 | about / contact / privacy — 양호 | 유지 |
| 공식 출처 | 다수 글에 링크 — 개선됨 | 미보강 글 추가 |
| **독창성** | 구조는 좋음, **warning 중복·FAQ 패턴** 리스크 | 본 문서 3절 조치 |
| 기술 | FAQ HTML, sitemap, robots — 정리됨 | 유지 |

**결론:** 구조와 콘텐츠 **형식**은 AdSense 신청 가능 수준입니다.  
**warning 전면 개별화 + FAQ 실패 케이스 보강**을 하면 「유사문서」 리스크가 크게 줄어듭니다.
